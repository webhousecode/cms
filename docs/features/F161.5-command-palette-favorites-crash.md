# F161.5 — Command-palette crash on a malformed favorite

**Status:** In progress · **Priority:** Critical (live prod crash for customer Sanne) · **Parent:** F161 (cms-admin — hærdning & polish)

## Symptom

On `webhouse.app/admin` (sanneandersen), the moment Sanne starts typing in the ⌘K command palette, the **whole admin app** crashes to the Next.js error boundary: *"Programfejl: Der opstod en klientsideundtagelse under indlæsning af webhouse.app"*. Reported with a screenshot on `/admin/content/sider-content`. Sanne-specific (her browser state), and triggered by *typing*, not by opening the palette.

## Root cause (confirmed by code inspection)

Favorites (F94) are persisted in `localStorage['cms-favorites']` and mirrored server-side (`/api/admin/user-state`). Three components read them directly and set React state **without validating the shape**:

- `components/command-palette.tsx` (own localStorage read + `cms:favorites-changed` event)
- `lib/hooks/use-favorites.ts` (localStorage + server hydration + event) — backs `favorites-list.tsx`
- `components/sidebar.tsx` (localStorage init + server hydration + event)

The `Favorite` type requires `label: string`, but persisted data can predate the current shape or carry a legacy entry with a missing/undefined `label` or `path`. When the palette has **no** query, the favorites filter short-circuits (`if (!q) return true`) and never touches `label` — so mount is fine. The instant a query is typed, the filter runs `f.label.toLowerCase()` (command-palette.tsx ~L347) on every favorite → `undefined.toLowerCase()` throws → React render fails → app-wide error boundary. The **exact same** crash lives in the Favorites page search box (`favorites-list.tsx` ~L61).

The server search route is NOT the cause: content titles fall back to slug (`String(... ?? doc.slug)`) and media titles fall back to `m.name`, so `item.label` from search results is always a string. The crash is purely the persisted-favorite data path.

## Fix (single source, validate at the trust boundary)

Add a pure, client-safe `sanitizeFavorites(raw: unknown): Favorite[]` in a new `lib/favorites.ts` (NOT `lib/user-state.ts`, which imports `fs` and would break the client bundle). It:

- returns `[]` for any non-array input (kills the `favorites.filter is not a function` variant too),
- drops elements that are not objects or have no usable `path`,
- coerces a missing/empty `label` to the `path`, and an unknown `type` to `"page"`,
- passes through `id/icon/collection/slug/addedAt` with safe fallbacks.

Apply it at **every** favorites read-boundary in the three components, for localStorage, server hydration, and the change event — so every downstream consumer can trust the `Favorite` shape. Keep a defensive `Highlight` guard in the palette as a render-leaf backstop.

## Verification

1. Unit test `lib/__tests__/favorites.test.ts` (node env): a malformed favorite (`{path}` with no `label`) survives with a string label; a non-array input → `[]`; `sanitizeFavorites(raw)` output is safe to `.toLowerCase()`. RED before the fix, GREEN after — this is the sealed harness gate.
2. `npx tsc --noEmit` clean across the touched files.
3. Deploy (push main → CI auto-deploy to webhouse.app).
4. Lens on webhouse.app (sanneandersen): open ⌘K, type a query, assert results render and no error-boundary crash.

## Non-goals

- Healing the malformed favorite in the server-side `user-state` store (the sanitizer makes it harmless at read-time; a later normalize-on-write pass can clean the stored data).
- Any change to how favorites are created or to the favorites UI.

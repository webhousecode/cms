## Design Decisions (confirmed)

| Decision | Choice | Rationale |
|----------|--------|-----------|
| **Framework** | @webhouse/cms + Next.js App Router | 100% dogfooding — proves the product |
| **Hosting** | Fly.io, region arn (Stockholm) | GitHub Pages can't serve Next.js SSR |
| **Versioning** | Automated per npm release | Standard: major.minor docs branches |
| **Language** | English primary, Danish via AI translate (F48) | International audience first |
| **Content authoring** | CMS admin UI via Claude Code agent | Markdown native in CMS, editable in editor |
| **Auto-generation** | 99% AI-generated, refreshed by scheduled agent | Human reviews, adds screenshots where needed |
| **HelpCard connection** | F116 articles = source of truth | Docs site expands articles into full pages |
| **API access** | Full markdown API for embedding in webhouse.app admin | MCP + REST endpoints for content |

## F116 HelpCard ↔ F31 Connection

### How it works

1. **Source of truth:** `packages/cms-admin/src/lib/help/articles.ts` — 10+ articles
2. **generate-help-docs.ts** reads articles and creates expanded doc pages:
   - HelpCard `body` → intro section
   - HelpCard `actions` → "Next Steps" section
   - Script adds: code examples, related docs links, screenshots placeholders
3. **`helpCardId` field** on docs links back to the in-app article
4. **`learnMorePath`** on HelpCard articles links to the docs page
5. **F115 CMS Help Chat** indexes both sources for search

### Data flow

```
F116 articles.ts  →  generate-help-docs.ts  →  docs site content/help/
       ↓                                              ↓
  HelpCard in admin                           docs.webhouse.app/help/{id}
       ↓                                              ↑
  learnMorePath  ─────────────────────────────────────┘
```

## Auto-Generation Pipeline

### 1. API Reference (`generate-api-docs.ts`)

Parses these sources:
- `packages/cms-admin/src/app/api/**/*.ts` → REST API endpoints
- `packages/cms/src/schema/types.ts` → TypeScript interfaces
- `packages/cms-cli/src/commands/*.ts` → CLI commands
- `packages/cms/src/build/*.ts` → Build pipeline phases
- OpenAPI spec if available

Output: One document per endpoint/type in `content/api-reference/`

### 2. Help Docs (`generate-help-docs.ts`)

Reads `HELP_ARTICLES` from F116, expands each into a full doc page:
- Title + expanded body
- Code examples (generated by AI)
- Related docs cross-links
- Screenshot placeholders (human fills in)

### 3. Changelog (`generate-changelog.ts`)

Reads git tags and generates changelog entries:
- `git tag --list 'v*'` → version list
- Commit messages between tags → content
- Breaking changes detected from commit prefixes

### 4. Content Refresh Agent

Claude Code agent (or scheduled skill) that:
1. Scans `packages/cms-admin/src/` for new API routes, components, settings
2. Compares against existing docs → identifies gaps
3. Generates/updates docs for new features
4. Runs on demand or after each npm version bump

```bash
# Manual trigger
npx tsx scripts/refresh-docs.ts

# Or via Claude Code skill
/docs-refresh
```

## Markdown API for webhouse.app Admin

The docs site exposes content via REST + MCP for embedding in CMS admin:

```
GET https://docs.webhouse.app/api/docs?slug=getting-started
→ { title, content (markdown), category, helpCardId }

GET https://docs.webhouse.app/api/docs/search?q=robots.txt
→ [{ slug, title, excerpt, score }]

GET https://docs.webhouse.app/api/help/{helpCardId}
→ { title, body, actions, fullContent }
```

This allows:
- CMS admin to fetch and display full doc content inline
- HelpCard "Learn more" to load content without leaving admin
- F107 Chat to reference docs in answers
- webhouse.app marketing site to embed docs sections

## Site Design

- **Layout:** Left sidebar (category nav) + main content + right TOC (heading nav)
- **Theme:** Dark by default (matches CMS admin), light toggle
- **Colors:** webhouse gold (#F7BB2E) accents on dark (#0D0D0D)
- **Typography:** System fonts (same as CMS admin)
- **Code blocks:** Shiki syntax highlighting, copy button
- **Search:** Full-text via CMS content API, ⌘K shortcut
- **Mobile:** Responsive, sidebar collapses to hamburger

## Versioning Strategy

- **`latest`** branch = current npm version docs (default)
- On each npm publish, tag docs content with version
- `/v/0.2.x/` prefix for archived versions (lazy — only create when breaking changes warrant it)
- Automated: `generate-changelog.ts` runs in publish workflow

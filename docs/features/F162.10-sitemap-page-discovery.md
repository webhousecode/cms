# F162.10 ‚Äî `--sitemap` page discovery (the gates ask the site for its page list)

## Motivation

Christian, sharply + rightly: the CMS obviously knows all its pages ‚Äî why does the coverage CLI require a hand-maintained `--pages` list? That hand-list is exactly how phantom/duplicate slugs crept into broberg (F162.8) and how the site's ~17 blog posts were silently never covered. The fix is a standard, not a broberg patch: **the site owns its page list (sitemap.xml) and the gates discover pages from it.**

## Why the site (not the CMS API) is the source

The CMS knows all CONTENT (docs) and default-locale urlPrefix/urlPattern ‚Äî the engine's `generateSitemap` already computes standard-site URLs. But a bespoke site like broberg does custom URL routing in its OWN app: category-based post URLs `/:category/:slug` (not declared in the CMS schema) and locale-segment localization `/flagskibe`‚Üî`/flagships` (in broberg's i18n, not the CMS). So for such sites the CMS alone cannot produce the correct URL set ‚Äî only the site's own router can. The universal answer that works for BOTH standard and bespoke sites: the site emits `sitemap.xml` from its own single source of truth, and the gates read it.

## Scope

1. **CLI (`@webhouse/cms-cli` 0.4.24):** `--sitemap <url>` on `coverage` + `check-editable`. Shared `resolveTargets()` (sitemap wins over url+pages) + `parseSitemapLocs()`; a sitemap index (all `<loc>` end in `*sitemap*.xml`) is followed one level. `--pages`/`--url` kept as manual override. Unit tests seal parsing + pages-mode.
2. **broberg `/sitemap.xml`:** extract `siteIndexGroups(locale)` from `renderSiteIndex` (the human /indeks). `src/sitemap.ts` flattens its hrefs across both locales into a valid urlset. Register `app.get('/sitemap.xml')` before the catch-all (it previously fell through to `renderGenericPage` ‚Äî an SEO bug too). Origin derived from the request (no hardcoded base).
3. **broberg gates:** switch `gate:editable` + `gate:coverage` to `--sitemap https://broberg.ai/sitemap.xml`; drop the static PUBLIC_ROUTES list. Classify any A.2 fields that scanning all pages surfaces (F162.9 overlaps here).
4. **Recipe + fleet:** docs/ai-guide/22 makes `--sitemap` the standard; sanne adopts (Next.js `app/sitemap.ts` via `cmsSitemap` likely already emits one) + migrates her bespoke A.1 to `cms check-editable`.

## Non-goals

- A CMS-side `/api/cms/urls` endpoint (a nice future for standard sites, but the sitemap approach is universal + needs no auth).
- Auto-deriving broberg's EN segment localization from the CMS (it lives in broberg's app; the sitemap already encodes it correctly via siteIndexGroups).

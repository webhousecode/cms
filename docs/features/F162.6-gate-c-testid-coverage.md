# F162.6 — Gate C: `cms check-testids` (Lens testID coverage)

## Motivation

Christian (2026-07-12): the deterministic gate tool that every website-repo installs should validate **all three** invariants of an editable, Lens-verifiable site, not two:

- **Gate A** (`cms coverage`) — every rendered CMS field is inline-editable (`data-cms-*`).
- **Gate B** (`cms check-text`) — no user-visible text lives outside the CMS.
- **Gate C** (`cms check-testids`) — every interactive element carries a `data-testid` so Lens can drive + assert it.

Today the testID-gap check lives only in the cardmem daemon (`POST 127.0.0.1:7475/lens/testid-gaps`, F086) — a site's CI cannot call it. Gate C moves the same invariant into the deterministic CLI so it runs in every site's CI next to A and B. One tool, three gates.

## Scope

- New `packages/cms-cli/src/commands/gate-c.ts` exporting a pure `findTestidGaps(ts, source, fileName)` (DI'd typescript, mirrors `gate-b.ts`).
- Interactive = JSX element that is one of `button|input|select|textarea|a[href]` OR carries an event handler attribute (`onClick|onChange|onInput|onSubmit|onKeyDown|...`). Flag when it has no `data-testid` attribute.
- New `packages/cms-cli/src/commands/check-testids.ts` — `cms check-testids --dir --allowlist --json`, non-zero exit on gaps; allowlist baseline identical to check-text.
- Register in `packages/cms-cli/src/index.ts`.
- Publish via the isolated `cms-cli-v*` tag path (bump + tag).

## Non-goals

- Not a replacement for the cardmem daemon's live testid-gaps (that stays for local/Lens flows); Gate C is the CI-runnable static equivalent.
- No auto-fixing / injecting testids.

## Rollout

- Ship in cms-cli, add a third `gate:testids` script + CI step to broberg, document for sanne. Each site baselines its current interactive elements once, then only NEW untagged elements fail.

## Verification

- Unit tests in `gate-c.test.ts` (the AC cases). Dogfood on broberg + confirm exit codes. Wire into broberg CI and prove the step runs green in Actions.

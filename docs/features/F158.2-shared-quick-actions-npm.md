# F158.2 — Shared client for cached quick-actions (one npm, three consumers)

## Motivation
The F158 server engine (cms-admin) already serves cached quick-action answers via `GET/POST /api/cms/chat/quick/:key`. But the CLIENT pattern that consumes it — *peek the cache → render the warm answer instantly, else stream and (optionally) warm-after-stream* — is hand-rolled in each surface:
- **cms-admin** — `handleSuggestionClick` in `components/chat/chat-interface.tsx` (inline `fetch`).
- **broberg.ai** — `sendQuick` in `src/client/admin-chat.tsx` (inline `fetch` + POST-back to warm).
- **sanne** — to be built.

Three copies of the same fetch contract = drift waiting to happen (rule-of-three). Christian's call (2026-07-09): "3 løsninger der bruger den samme teknik kalder på en fælles npm." One shared source; each app keeps its own rendering.

## Scope
Extract the framework-agnostic client contract into ONE shared npm and adopt it in cms-admin + broberg. Sanne adopts the same package in a sibling task (sanne repo, coordinated by intercom). The server endpoint is unchanged.

Exports (framework-agnostic, zero React/Preact deps):
- `peekQuickAction({ baseUrl, siteId, key, headers?, fetchImpl? }) → Promise<{ cached: boolean; markdown: string; cachedAt: number }>` — GET the cache. Never throws; a network error resolves to `{cached:false}` so callers always fall back to streaming.
- `warmQuickAction({ baseUrl, siteId, key, markdown, headers?, fetchImpl? }) → Promise<void>` — POST a finished answer back (broberg does this after a cold stream; cms-admin relies on the server's lazy-regen so it can skip this).
- `QUICK_ACTION_KEYS` / a typed `QuickActionKey` union kept in sync with the server's `quick-actions.ts` (overview | drafts | site-info | capabilities).

Auth is the CALLER's responsibility (passed via `headers`): cms-admin uses same-origin cookies (no header), broberg's relay injects the service token upstream, sanne passes its `CMS_API_TOKEN`. The package does not own auth.

## Home (decide at build time — reuse-first)
1. **Preferred:** fold into the in-progress **F149 `@webhouse/*` Web App SDK** if its shape fits (one SDK surface for site clients). Check F149 first.
2. **Else:** a small dedicated `@webhouse/cms-quick-actions` package in `packages/` (same publish/OIDC setup as `cms-inline-edit`), added to `.github/workflows/publish.yml` + the npmjs trusted-publisher step.

Do NOT hand-roll a second copy anywhere once the package exists.

## Non-goals
- No change to the server engine / endpoint (F158.1, done).
- Not a UI component — each app renders the markdown its own way (React/Preact). The package is fetch-logic only.
- Not the sanne integration itself (sibling task in the sanne repo).

## Rollout
1. Land the package (or SDK addition) with unit tests + publish.
2. cms-admin: replace the inline `fetch` in `handleSuggestionClick` with `peekQuickAction`. Lens re-verify (warm → instant).
3. broberg: replace `sendQuick`'s inline `fetch`/POST-back with `peekQuickAction` + `warmQuickAction`. Lens re-verify.
4. Intercom sanne with the package name + endpoint contract; they adopt + card it in their project.

## Verification
- Package: unit tests for peek/warm (happy path, network-error → cached:false, empty markdown no-op).
- cms-admin + broberg: Lens re-run of the warm quick-action click still renders instantly (no regression) after swapping to the shared package.
- `npm pack` / publish succeeds; consumers pin the published version.

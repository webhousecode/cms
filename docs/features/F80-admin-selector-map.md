# F80 — CMS Admin Selector Map

> Auto-generated map of stable `data-testid` selectors for every CMS admin UI element, enabling reliable Playwright tests for content editing, site management, and roundtrip verification.

## Problem

Today there are zero `data-testid` attributes in the CMS admin codebase. Playwright tests (F65, F20) must rely on fragile CSS selectors or `getByText()` which break when UI text or layout changes. Writing automated tests for site templates — like verifying that a new static site's content is editable and persists through a build cycle — requires manually inspecting the DOM to find selectors. This makes E2E testing slow to write, brittle to maintain, and impossible to generate automatically.

## Solution

1. Add `data-testid` attributes to every interactive element in CMS admin (fields, buttons, nav, dialogs)
2. Auto-generate a `selector-map.json` manifest that maps logical names to selectors
3. Ship Playwright test fixtures and helpers that use the map for common workflows
4. Provide a CLI command to dump/refresh the selector map

## Technical Design

### data-testid Convention

Every interactive element gets a `data-testid` following this pattern:

```
{context}-{element-type}[-{identifier}]
```

Examples:
```
nav-link-sites              # Sidebar "Sites" link
nav-link-content-pages      # Sidebar "Pages" under Content
site-card-meridian-studio   # Site card on /admin/sites
field-text-title            # Text input for "title" field
field-richtext-content      # Richtext editor for "content" field
field-image-coverImage      # Image picker for "coverImage"
field-array-images          # Array field container for "images"
btn-save                    # Save button
btn-publish                 # Publish button
btn-unpublish               # Unpublish button
btn-delete                  # Delete button
btn-create                  # Create new document button
dialog-confirm              # Confirmation dialog
dialog-confirm-yes          # Confirm "Yes" button
dialog-confirm-no           # Confirm "No" button
collection-list-posts       # Document list for "posts" collection
collection-item-hello-world # Individual document row
site-switcher               # Site switcher dropdown
org-switcher                # Org switcher dropdown
```

### Auto-generated Field Selectors

Field `data-testid` attributes are generated from the CMS config schema at render time. In the field renderer component:

```typescript
// packages/cms-admin/src/components/fields/field-renderer.tsx
function FieldRenderer({ field, value, onChange }: FieldRendererProps) {
  const testId = `field-${field.type}-${field.name}`;
  return (
    <div data-testid={testId}>
      {/* field component */}
    </div>
  );
}
```

### Selector Map Manifest (`selector-map.json`)

Generated by crawling the admin at runtime or statically from config:

```typescript
interface SelectorMap {
  version: string;
  generated: string;
  navigation: Record<string, string>;    // logical name → data-testid
  actions: Record<string, string>;       // buttons, links
  fields: Record<string, {              // per-collection field selectors
    collection: string;
    field: string;
    type: string;
    testId: string;
    inputSelector: string;              // the actual input/textarea/div within
  }>;
  dialogs: Record<string, string>;
}
```

### Playwright Test Helpers (`packages/cms-admin/tests/helpers/`)

```typescript
// helpers/selectors.ts
import selectorMap from '../selector-map.json';

export function field(name: string) {
  return `[data-testid="field-${name}"]`;
}

export function fieldInput(collection: string, fieldName: string) {
  const entry = selectorMap.fields[`${collection}.${fieldName}`];
  return entry?.inputSelector ?? `[data-testid="field-${fieldName}"] input`;
}

export function btn(name: string) {
  return `[data-testid="btn-${name}"]`;
```

### Example Test Fixtures

```typescript
// tests/site-roundtrip.spec.ts
import { test, expect } from '@playwright/test';
import { loginAsAdmin, switchToSite, openDocument, editTextField, saveDocument } from './helpers/workflows';

test('edit title in admin → rebuild → verify in HTML', async ({ page }) => {
  await loginAsAdmin(page);
  await switchToSite(page, 'Elina Voss Portfolio');
  await openDocument(page, 'projects', 'editorial-styling');

  // Reload and verify persistence
```

### CLI Command

```bash
npx cms selector-map [--site <siteId>] [--output selector-map.json]
```

Reads `cms.config.ts` for the active site and generates the field selector map without requiring a running server.

## Implementation Steps

1. Define `data-testid` naming convention (this document)
2. Add `data-testid` to all interactive elements in CMS admin:
   - Field renderer components (auto from field name/type)
   - Navigation sidebar links
   - Action buttons (save, publish, delete, create)
   - Dialogs and dialog buttons
   - Site/org switcher
   - Collection list items
   - Site cards
3. Create `selector-map.json` generator (reads cms.config.ts → outputs map)
4. Create Playwright test helpers (`helpers/selectors.ts`, `helpers/workflows.ts`)
5. Write example test fixtures for common workflows
6. Add `cms selector-map` CLI command
7. Document the convention for future contributors

## Dependencies

- None strictly required, but benefits from:
  - F65 (Agent Pipeline E2E Tests) — reuses mock LLM fixture pattern
  - F20 (Visual Testing) — shares Playwright infrastructure
  - F79 (Site Config Validator) — validates config before generating map

## Effort Estimate

**Medium** — 4-5 days. Most work is adding `data-testid` attributes across the admin UI components (mechanical but thorough). Test helpers and CLI are straightforward once the attributes exist.

# F162.7 — `cms check-editable` (reusable page-level visible-text scan / Gate A.1)

## Motivation

Sanne's site proved a critical gap in Gate A: `cms coverage` (A.2) is **schema-driven + union-per-document** — a field editable on ANY page counts as covered. A field rendered NON-editably on a specific page slips through. Sanne closed it with a bespoke 115-line `site/scripts/inline-edit-coverage.mjs` (a DOM visible-text scan). Per the fleet **reuse-first** HARD RULE, copying that script into broberg (and every future site) = drift. Instead we bake the proven scan into the shared tool `@webhouse/cms-cli` as `cms check-editable`, so broberg, sanne, and all future sites share ONE implementation. This is also the enabler for the F162.4 fælles-standard (a recipe that ships a shared command, not a copied script).

## What it does (ported from sanne's proven `scan()`)

For each `--pages` route: `fetch()` the live SSR HTML from `--url` base, parse with `node-html-parser`, look inside `<main>`, and flag every **content-leaf** element with visible prose that a visitor sees but cannot inline-edit.

- `CONTENT_SEL = h1,h2,h3,h4,h5,h6,p,li,blockquote,figcaption,dd`
- `NOISE_SEL = nav,header,footer,button,form,input,select,textarea,[role=button],inline-edit's own UI attrs,script,style,svg`
- Gap iff: text length >= 3 AND not `el.closest('[data-cms-field]')` AND not `el.closest(NOISE_SEL)` AND not a container (`el.querySelector('h1..h6,p,li,[data-cms-field]')` is null) AND text doesn't match an `--ignore-text` substring.
- Strict/absolute: **0 gaps required**, no baseline (unlike A.2's no-new-gaps). Intentional-exclude (token fields like `{år} års erfaring`) reported separately, never as gaps.

## Non-goals

- Schema coverage (that stays `cms coverage` / A.2).
- Source-AST hardcoded-string detection (that stays `cms check-text` / Gate B).
- Browser/Playwright — pure `node-html-parser`, headless, no auth.

## Architecture

- New command file `packages/cms-cli/src/commands/check-editable.ts` + registration in `index.ts`.
- `node-html-parser` becomes a direct (small, no-native) dependency of cms-cli so the command works out of the box (no per-site install), unlike the optional `@broberg/lens-engine` peer used by `coverage`.
- Pure `scan(html, opts)` function exported + unit-tested against HTML fixtures (no network in tests).
- Options: `--url`, `--pages`, `--ignore-text`, `--content-sel?` (default proven set), `--json`.

## Rollout

Bump cms-cli patch, publish via the isolated `cms-cli-v<ver>` tag path (NOT the fleet workflow_dispatch — that would drag held-back token-safe packages out). Consumers upgrade `@webhouse/cms-cli` and add a one-line `gate:editable` script.

## Verification

`cd packages/cms-cli && npx vitest run` green; `node dist/index.js check-editable --url https://broberg.ai --pages /` runs against live broberg and reports real gaps (drives F162.8 burn-down).

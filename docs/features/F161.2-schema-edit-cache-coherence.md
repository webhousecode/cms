# F161.2 — Schema-edits surfacer uden machine-restart

> Parent: F161. Kilde: memory schema-drift-add-to-schema-dataloss.

## Motivation

Redigerer man et collection-schema i admin-UI'et, ses ændringen først efter en Fly-machine-restart. Årsag: den kompilerede-config-cache (site-pool) lever "forever" i prod, og middleware + route-handlers er separate module-instanser. En schema-write i én instans ses ikke i den anden før cold start. Papercut for enhver kunde der rører schema.

## Scope

- En schema-write (PUT/POST/DELETE `/api/schema/*`) skal surface på NÆSTE request i BÅDE middleware og route-handler uden restart.
- Via delt signal (fil-mtime-reload på den kompilerede config, som `loadRegistry` allerede gør) ELLER eksplicit cross-instans-invalidering.

## Non-goals

- config-writer-rewrite (separat hærdning).
- Multi-machine (F152).

## Arkitektur-skitse

1. **Verificér nuværende tilstand først**: er `invalidateActiveSite()` nok, eller overlever site-pool på tværs af instanser? 2. Tilføj mtime-baseret reload til site-pool-cachen (spejler `site-registry.ts`-fixet), så en config-write i én instans læses i alle på næste request. 3. Seal med test: write via route → middleware ser det næste request. Cache forbliver keyed (orgId, siteId) — ingen cross-tenant-leak.

## Afhængigheder

site-pool.ts, config-writer.ts, schema-routes, site-registry.ts (mtime-mønster).

Kubernetes Secrets Management Done Right

Feb 28, 2026 devops

The Default is Broken

Kubernetes Secrets are base64-encoded, not encrypted. Anyone with read access to the API server can decode them trivially. Yet many teams treat kubectl create secret as their entire secrets strategy. This is a disaster waiting to happen.

Encryption at Rest

The first step is enabling encryption at rest for etcd. Kubernetes supports this natively through EncryptionConfiguration, but the key management is on you. For production clusters, integrate with a KMS provider — AWS KMS, GCP Cloud KMS, or HashiCorp Vault's transit engine.

The critical insight is that encryption at rest protects against etcd compromise, but not against someone with RBAC access to read Secrets through the API. You need both layers.

External Secrets Operators

The modern approach is to store secrets in an external vault (HashiCorp Vault, AWS Secrets Manager, or even 1Password) and sync them into Kubernetes automatically. The External Secrets Operator watches for ExternalSecret resources and creates native Kubernetes Secrets from your vault's data.

This gives you audit logging, rotation, and access control that Kubernetes alone cannot provide. The tradeoff is operational complexity — you now have a critical dependency on your vault being available.

Sealed Secrets for GitOps

If your workflow is GitOps-based, Bitnami's Sealed Secrets let you encrypt secrets with a cluster-specific key and commit the ciphertext to Git. The controller in the cluster decrypts them. This means your Git repository contains the full desired state, including secrets, without exposing plaintext.
